# SteadyOps Agent 能力评审：定位、缺陷与内置工具建议

> 评审日期：2026-07-19
> 评审视角：从「Agent」角度审视——它算不算 Agent、对非研发运维人员有哪些不便利、还应内置哪些工具
> 说明：本文聚焦 **Agent 能力与易用性**；产品自身的安全/可靠性代码缺陷另见 [运维层面缺陷审查报告](ops-defects-review.md)。

---

## 一、它能称作 Agent 吗？

**能，而且是一个架构上相当标准的 Agent。** 按业界对 Agent 的通用判据逐项核对：

| 判据 | SteadyOps 现状 | 证据 |
|------|---------------|------|
| LLM 自主决策 | LangGraph ReAct 循环：思考 → 选工具 → 观察 → 再推理 | `agent/langchain/chat_graph.py` |
| 工具使用 | 23 个内置 Tool，覆盖查询/日志/文件/执行/巡检/告警/诊断 | `agent/langchain/tools.py:181` |
| 环境感知 | System prompt 动态注入主机清单、服务清单、长期记忆 | `agent/langchain/context_builder.py:24` |
| 记忆 | 短期：LangGraph Checkpointer 持久化对话；长期：知识库三分类条目自动抽取 + prompt 注入 | `agent/store/knowledge.py`、`agent/langchain/memory_extractor.py` |
| 行动能力 | 可通过 SSH 重启服务、读写远程文件、执行命令 | `agent/remediation/` |
| 安全边界 | 写操作人工确认门禁、重启冷却、命令风险分级 | `agent/executor/command_policy.py` |
| 主动运行 | 后台 MonitorLoop 按间隔持续巡检并产生告警 | `agent/monitor/loop.py` |

**更准确的定位是「对话式运维 Copilot + 巡检告警 Agent」**，而不是全自主 Agent。差距在三个地方：

1. **被动为主、主动为辅。** AI 诊断能力（4 个风险评估 playbook）只能等用户在对话里提问才触发，监控循环不会调用它们；它不会自己发现疑点然后主动深挖。
2. **没有目标驱动的规划。** 只有单轮 ReAct 反应式循环，没有「给定一个目标（如：把磁盘占用降到 80% 以下）→ 拆解步骤 → 跨多轮执行 → 验证结果」的能力。
3. **没有编排能力。** 无定时任务、无跨主机批量、无多步骤 runbook，所有动作都是「一问一答一工具」。

一句话结论：**叫它 Agent 名副其实，但它是「反应式 Agent」，离「自主运维 Agent」还差主动性、编排和自愈三层能力。**

## 二、面向非研发运维人员的缺陷与不便利

以下按对日常使用的影响排序。非研发人员的核心诉求是：**少懂概念、少做确认、告警别吵也别漏、常见问题一键搞定**。

### 2.1 监控告警太薄——「看得见服务死活，看不见系统趋势」

- **巡检规则只有 4 类**：服务未运行(P0)、HTTP 健康检查失败(P1，连失败 3 次升 P0)、日志 ERROR 激增、4 个硬编码日志关键字（`agent/incident/rules.py:14-65`）。
- **主机 CPU/内存/磁盘完全不参与巡检**——`get_host_metrics` 只能对话里临时查，且是瞬时值、**不落库、无时序**。非研发人员最常吃的就是「磁盘写满」「内存缓慢泄漏」这类渐进故障，当前体系只能在服务挂了之后才发现。→ 桌面端主机资源面板规划见 [host-metrics-panel-plan.md](host-metrics-panel-plan.md)。
- **主机整机不可达（SSH 连不上）不产生任何告警**——最需要通知的场景是沉默的。
- **告警无收敛体系**：去重靠进程内存 set，进程重启后历史告警全部重发一遍；故障修复后复发反而不再告警；无静默（silence）、无恢复通知、无负责人认领（`agent/monitor/loop.py:33,79-93`）。
- **告警渠道只有飞书**，没有邮件 / 通用 webhook / 企业微信 / 钉钉，不用飞书的团队等于没有通知。

### 2.2 主动性断层——「好东西只能问出来，不会自己送上门」

- 4 个评估 playbook（OOM/CPU/假活/假健康）质量不错，但**只能 AI 对话触发**：无定时执行、无 UI 一键按钮、监控循环不调用（`agent/playbooks/`，注册于 `tools.py:754-757`）。非研发人员根本不知道该什么时候问、怎么问。
- 没有**巡检日报/周报**。现有的「问题报告」是事件驱动的个案归档（`agent/ops/`），不能回答「上周整体运行情况怎么样」这种管理层常问的问题。
- 没有日志关键词**订阅**机制——想盯某个特定报错（如某业务的 `TimeoutException`）只能每次手动让 AI 去查。

### 2.3 操作层不便利——「一次只能干一件事，确认还很多」

- **无跨主机批量**：所有工具只接受单个 host_id，给 10 台机器查磁盘要 ReAct 循环 10 次，慢且费 token（`tools.py` 全体工具签名）。
- **写操作确认只能单个来**：system prompt 强制「每次只提议一个待确认操作」，UI 一次只显示一个确认框，新事件直接覆盖旧的（`context_builder.py:61`、`desktop/pages/../widgets/chat_panel.py:229-237`）。批量变更窗口期操作很折磨。
- **确认有审美疲劳风险，且门禁本身可被绕过**：命令策略是黑名单思路（默认放行、命中模式才确认），`python3 -c "..."`、`sed -i`、`find -delete` 等写操作**不经确认直接执行**（`command_policy.py:18-79`）；写路径白名单 `is_path_allowed` 定义了却全代码库无调用，是死配置（`write_policy.py:30`）。对非研发人员来说这意味着「确认弹窗给了虚假的安全感」。
- **pending 操作纯内存存储**：进程重启即丢失，30 分钟 TTL 过期后确认返回 404（`remediation/pending_writes.py:27-35`）。
- **重启意图靠正则前置拦截**（`chat_graph.py:94`），绕过了 LLM 理解，口语化表达（「把 xx 拉起来」「xx 挂了帮我弄一下」）可能识别不到。

### 2.4 知识与上下文——「记性是 prompt 撑的，没有真检索」

- 长期记忆是**纯 prompt 注入**，无向量检索/RAG；条目多了以后靠「最近 20 条 + 2000 token 裁剪」，老知识会被挤掉（`knowledge.py:186`）。
- 自动抽取的记忆**直接入库无需人工确认**，与 marker 建议需确认的路径并存，小模型抽错事实会污染后续所有对话（`memory_extractor.py:83`）。
- 没有服务拓扑/依赖关系模型——「A 服务挂了会影响谁」Agent 回答不了，而这恰恰是告警时非研发人员最需要的判断依据。

### 2.5 使用门槛——「还是得懂点研发概念」

- 用户需要理解 host_id、service_id 这些标识概念（虽有 IP/模糊匹配回落，`settings.py:87`），对话里说「生产那台跑订单的机器」未必能解析。
- 可信度标签、pending_confirm 等机制设计得好，但**实际体验强依赖 LLM 的 ReAct 能力**；内网用 Ollama 小模型时，多步工具调用的成功率会显著下降，而文档没有给出模型能力的下限建议。
- 飞书机器人侧**只读且关键词拦截**，无法确认写操作——人不在电脑前时（恰是告警高发时段）只能干看着。

## 三、建议内置的工具与能力（按优先级）

原则：优先补「非研发人员每天都会用到、现在只能手动做」的事；每条标注实现成本。

### P0：告警与可观测性地基（不补这些，Agent 越强越危险）

| # | 建议 | 解决什么 | 成本 |
|---|------|---------|------|
| 1 | **主机指标阈值巡检 + 时序落库**：CPU/内存/磁盘/load 每轮采集存 SQLite，超阈值告警；新增 `get_metrics_history` 工具供 AI 看趋势 | 渐进故障（磁盘满、内存泄漏）提前发现；AI 诊断有历史数据可依 | 中 |
| 2 | **告警生命周期管理**：firing/resolved 状态机、恢复通知、告警静默（按服务/时长）、去重加 TTL | 「重启重发、复发不报」的反直觉行为；告警噪音 | 中 |
| 3 | **主机不可达告警**：SSH connect 失败升级为 incident | 整机宕机沉默 | 低 |
| 4 | **通用 webhook + 邮件告警渠道**：一个 POST 模板即可覆盖企业微信/钉钉/飞书自定义机器人 | 非飞书团队无通知 | 低 |
| 5 | **命令策略改白名单思路**：默认 confirm、显式只读模式才 allow；识别 `python/perl -c`、`sed -i`、`find -delete` 等绕过模式；启用写路径白名单（现有死代码） | 确认门禁形同虚设的绕过面 | 中 |

### P1：日常运维提效工具（每天会用到的）

| # | 建议 | 典型对话 | 成本 |
|---|------|---------|------|
| 6 | **定时任务编排**（内置 cron 调度器 + `schedule_task` 工具）：定时跑巡检/清理/采集，结果可推送 | 「每天凌晨 4 点清理 7 天前的日志」「每小时检查一下证书有效期」 | 中 |
| 7 | **批量主机操作**：`run_on_hosts(hosts=[...], cmd)` 并行 fan-out + 结果汇总 | 「看看所有机器磁盘还剩多少」 | 中 |
| 8 | **磁盘清理助手** `cleanup_disk`：定位大目录/旧日志/孤儿 docker 镜像，给清单 → 确认后清理 | 「磁盘 90% 了，帮我看看能清什么」 | 中 |
| 9 | **证书/域名到期检查** `check_cert_expiry` | 「查查各服务 HTTPS 证书什么时候到期」 | 低 |
| 10 | **日志关键词订阅**：用户自定义关键字 + 服务 → 纳入巡检规则，替代现在的 4 个硬编码 | 「帮我盯着 road_control 的 TimeoutException」 | 低 |
| 11 | **巡检日报/周报生成**：定时聚合指标趋势 + 告警统计 + AI 摘要，推送飞书 | 「生成本周运维周报」 | 中 |
| 12 | **playbook 开放手动/定时触发**：4 个评估 playbook 加 UI 一键运行按钮和定时开关，不只靠对话 | 巡检页点一下「评估 OOM 风险」 | 低 |
| 13 | **批量确认模式**：同一会话内多个低危写操作可「全部确认」，高危仍逐个 | 变更窗口批量执行 | 低 |

### P2：进阶诊断与自治（让 Agent 更像 Agent）

| # | 建议 | 价值 | 成本 |
|---|------|------|------|
| 14 | **服务依赖登记**：配置里加 `depends_on` 字段（先手工登记，不做自动发现），告警时 AI 可答影响面 | 告警时快速判断上下游影响 | 低 |
| 15 | **配置备份与 diff**：`backup_config` / `diff_config`，写文件前自动备份，确认弹窗展示与原文 diff | 改配置可回滚、变更有迹可循 | 中 |
| 16 | **用户自定义 runbook**：YAML 声明式多步骤流程（探测 → 判断 → 动作 → 验证），AI 按名调用，复用现有确认门禁 | 把团队处置 SOP 固化给 Agent 执行 | 高 |
| 17 | **网络连通性探测**：`ping` / TCP 端口 / DNS 解析检查工具 | 「这台机器访问数据库通不通」类问题 | 低 |
| 18 | **知识库升级 RAG**：条目向量化 + 检索注入，自动抽取加人工确认开关 | 长期记忆不被裁剪掉、减少错误记忆污染 | 高 |
| 19 | **只读 SQL 查询工具**（白名单 SELECT、独立账号、行数上限） | 运维常需查业务表验证数据 | 中 |
| 20 | **飞书侧写操作确认**：卡片按钮回调完成确认，而非只读 | 不在电脑前也能处置告警 | 中 |

### 不建议做的

- **全自动自愈/自动拉起服务**：在命令策略、告警状态机、依赖关系都健全之前，自动重启会把问题埋得更深（现有冷却限制是有意为之，保留）。
- **拓扑自动发现**：投入产出比低，手工登记 `depends_on` 已能覆盖 80% 场景。

## 四、总结

- **Agent 之名：成立。** 标准 ReAct + 23 工具 + 记忆 + 人工门禁，是合规的对话式运维 Agent。
- **最大短板不是 AI，是运维基本功**：监控无指标时序、告警无生命周期、操作无编排——这三层补不上，再强的模型也只是「回答得更漂亮的告警器」。
- **落地顺序建议**：先做 P0 的 1/2/3/5（可观测性 + 安全地基），再做 P1 的 6/7/8/10/12（定时编排、批量、清理、订阅、playbook 开放），这批做完，非研发人员 80% 的日常手动操作都能交给对话完成。
