# 主机资源监控面板规划（桌面 / exe）

> 规划日期：2026-07-20  
> 修订：2026-07-20 — 确定纵向三段布局，放弃左右分栏；服务概览改为矮条摘要  
> 范围：**仅桌面端（PySide6 / 打包 exe）**，本期不改 Web 端  
> 背景：针对 [Agent 能力评审 §2.1](agent-capability-review.md#21-监控告警太薄看得见服务死活看不见系统趋势)——主机 CPU/内存/磁盘不参与巡检、只能对话里瞬时查询。  
> 目标：在概览页增加主机资源可视化，让非研发运维「看得见系统现状」，并为后续阈值巡检与时序落库打 UI 基础。

## 一、问题与目标

### 现状

| 能力 | 现状 |
|------|------|
| 主机指标采集 | 后端已有 `SshExecutor.get_metrics()`，LangChain 工具 `get_host_metrics` 可用 |
| 桌面 UI | 无主机资源展示；无图表/进度条/实时轮询 |
| 巡检 | 仅服务状态 / HTTP / 日志，**不含** CPU/内存/磁盘 |
| 时序 | 瞬时值，不落库，无趋势 |

### 本期目标（exe 概览页）

- **中部资源条**：一眼看到 CPU / 内存 / 磁盘使用率
- **可展开详情**：点开查看 CPU Top5、内存 Top5、磁盘挂载点等
- 磁盘以**挂载点容量**为主（见第四节）
- **对话区保持全宽**，不因资源监控变窄

### 明确不做（本期）

- Web 端任何改动
- 独立「主机监控」导航页
- 概览页左右分栏（资源 vs 对话并排）——默认 1280×800 下对话会挤、纵向也塞不下 Top5

### 后续目标（对齐评审 P0）

- 指标时序落库 + 超阈值告警进巡检
- 大目录占用 Top5、写 I/O Top5（可选）
- 与 AI 对话联动（点击进程预填分析意图）

## 二、页面布局（已定）

### 2.1 方案结论

默认窗口约 **1280×800**（导航 200px）。左右分栏会压缩对话宽度，且左侧纵向塞不下「三表盘 + Top5 + 多挂载点」，美观与齐全难以兼顾。

**已定：概览页纵向三段 + 资源条可展开。**

| 方案 | 说明 | 结论 |
|------|------|------|
| A. 塞进全局 Sidebar | 200px 仅导航，放不下详情 | ❌ |
| B. 左右分栏（资源 \| 对话） | 对话变窄、密度冲突 | ❌ 否决 |
| C. 独立主机监控页 | 多一次点击 | 二期可选 |
| **D. 上服务 / 中资源条 / 下对话** | 对话全宽；详情按需展开 | ✅ **已定** |

### 2.2 布局示意

```
┌──────────┬──────────────────────────────────────────────┐
│ 导航栏   │ 【上】服务概览（收窄：矮条，非大卡片）            │
│ 200px    │     正常 N · 异常 N · 停用 N  |  [告警]        │
│          ├──────────────────────────────────────────────┤
│          │ 【中】主机资源条（默认折叠高度，横排一眼扫完）   │
│          │     CPU ████ 42%   内存 ████ 68%   磁盘 ████ 81%│
│          │     [展开详情 ▾]                               │
│          │     ┌─ 展开后 ─────────────────────────────┐  │
│          │     │ Tab：CPU Top5 | 内存 Top5 | 磁盘详情  │  │
│          │     │ （进程表 / 挂载点列表）                 │  │
│          │     └──────────────────────────────────────┘  │
│          ├──────────────────────────────────────────────┤
│          │ 【下】对话区（主区域，全宽，stretch 占满剩余）   │
└──────────┴──────────────────────────────────────────────┘
```

垂直比例建议（可微调）：

| 区域 | 高度策略 |
|------|----------|
| 上：服务概览 | **收窄**：约 40–56px 单行摘要条（见 2.4），把高度让给资源条与对话 |
| 中：资源条（收起） | 约 56–72px 单行 |
| 中：资源条（展开） | 收起高度 + 详情区约 180–240px；超出可内部滚动 |
| 下：对话 | `stretch=1`，吃掉剩余高度 |

### 2.4 服务概览收窄（已定）

现有「服务概览」三块大 `StatCard` + 告警按钮占高偏多，挤占中部资源与下部对话。实现时改为**矮条摘要**，信息保留、视觉降权：

| 现有 | 改为 |
|------|------|
| 三张大 StatCard（正常 / 异常 / 停用） | 同一行文字或小徽章：`正常 N · 异常 N · 停用 N`，异常用强调色 |
| 独立大块「告警记录」按钮 | 同行右侧小按钮 / 链接 |
| 标题 + host_hint 单独占一行 | 可并入同一矮条左侧（如「服务 · 主机名」）或省略独立 section 标题 |

交互不变：点击「正常 / 异常 / 停用」仍可进入对应服务列表；点击告警仍跳转告警页。

原则：**上条只做状态入口，中条做主机资源，下区留给对话。**

### 2.3 交互约定

- **绑定 TopBar 当前主机**：切换主机时刷新资源条；展开态可保留或自动收起（实现时二选一，建议保留展开态）
- **无主机**：资源条显示空态（「请先在顶部选择主机」），不发起轮询
- **离开概览页**：停止轮询；收起详情可选
- **展开 / 收起**：点击「展开详情」或资源条本身；再点收起。默认**收起**，保证对话区优先
- **样式**：复用 `Card`、进度条、QSS 主题 token；MVP 不引入图表库
- **范围**：只改 `agent/desktop/` 及共享后端采集；不动 Web

涉及文件（实现时）：

- `agent/desktop/pages/home_page.py` — 概览页插入中部资源条
- 新建 `agent/desktop/widgets/host_metrics_bar.py` — 资源条 + 可展开详情
- `agent/services/agent_service.py` — 封装 metrics / top 进程查询
- `agent/executor/ssh.py` — 扩展多挂载点、进程 Top5 采集

## 三、中部资源条内容结构

### 3.1 默认态（收起）：整机使用率横排

| 指标 | 展示形式 | 刷新频率 |
|------|----------|----------|
| CPU | 短进度条 + 百分比；可选附 load | 3–5 秒 |
| 内存 | 短进度条 + 已用/总量（如 `6.2G / 16G`） | 3–5 秒 |
| 磁盘 | 短进度条 + **主分区或最满分区** 百分比；旁注「N 个挂载点」 | 15–30 秒 |

**阈值着色（UI 与后续告警共用）：**

| 区间 | 语义 | 建议色 |
|------|------|--------|
| &lt; 70% | 正常 | 绿色 / 默认 |
| 70% – 85% | 警告 | 橙色 |
| ≥ 85% | 危险 | 红色 |

任一指标进入警告/危险时，对应条高亮，并可用小红点提示「有详情可看」。

> 现有 `get_metrics` 仅采集根分区 `/`，实现时需扩展为**相关挂载点**；收起态磁盘优先展示**使用率最高**的挂载点（或 `/`），展开态再列全部。

### 3.2 展开态：详情 Tab

默认进入展开后展示三个 Tab（或等价分段控件）：

| Tab | 内容 | 列 / 说明 | 刷新 |
|-----|------|-----------|------|
| CPU Top5 | 进程表 | PID / 进程名 / CPU% / 用户 | 展开时与资源条同频 3–5 秒 |
| 内存 Top5 | 进程表 | PID / 进程名 / RSS 或内存% / 用户 | 同上 |
| 磁盘详情 | 挂载点列表 | 路径 + 进度条 + 已用/总量/% | 15–30 秒 |

**交互（可选，二期）：**

- 点击进程行 → 对话区预填「分析 PID xxx 的 CPU/内存占用」
- 不提供杀进程等写操作

**收起时：** 停止 Top5 专用请求亦可（仅保留整机三指标轮询），减轻 SSH；再次展开后立即采一次。

### 3.3 磁盘补充（展开态内）

见第四节。MVP 挂载点列表即可；大目录 Top 放在磁盘 Tab 底部，低频或手动「刷新目录」。

## 四、磁盘监控策略

磁盘与 CPU/内存故障模式不同，**不应**用同一套「进程 Disk Top5」当主内容。

### 4.1 场景对照

| 场景 | 应看什么 | 「进程 Disk Top5」是否有用 |
|------|----------|---------------------------|
| 磁盘写满（渐进故障） | **挂载点容量 % + 增长趋势** | 几乎无用 |
| 谁占用了大量空间 | **目录占用 TopN**（`du`） | 进程列表帮不上忙 |
| 谁在狂写磁盘 | **写 I/O 速率 Top** | 有用，但是另一类指标 |

### 4.2 分层建议

| 层级 | 内容 | 刷新 | 阶段 |
|------|------|------|------|
| **L1** | 收起：最满分区%；展开：全部挂载点 | 15–30 秒 | **MVP** |
| **L2** | 大目录占用 Top5 | 手动或 ≥ 5 分钟 | 紧接着 |
| **L3** | 写 I/O Top5 | 5–10 秒 | 二期可选 |
| **L4** | 容量时序 + sparkline + 阈值告警 | 落库后 MonitorLoop | 对齐评审 P0 #1 |

**结论：**

- 磁盘**主视图** = 挂载点容量
- 磁盘**辅视图** = 大目录 Top5（低频 / 手动）
- **不要**用「进程 Disk% Top5」冒充磁盘监控

## 五、数据刷新与性能

每次采样均通过 SSH，单次约 **0.5–1 秒+**。

| 数据类型 | 建议间隔 | 原因 |
|----------|----------|------|
| CPU / 内存（收起条） | 3–5 秒 | 在概览页即轮询 |
| 进程 Top5 | 仅展开时 3–5 秒 | 收起不采，省 SSH |
| 挂载点容量 | 15–30 秒 | 变化慢 |
| 大目录 Top5 | 手动或 ≥ 5 分钟 | `du` 开销大 |
| 离开概览 / 切换主机 | 停轮询；取消未完成 AsyncCall | 避免堆积 |

**实现要点：**

- `AsyncCall` + `QTimer`，只保留最新一次结果
- 切主机递增 token，旧回调丢弃
- SSH 失败：资源条显示「主机不可达」，不 silent fail
